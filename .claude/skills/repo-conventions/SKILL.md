# Repository conventions

Hand-edit this file after installing the bundle. The bundle ships with the defaults below; replace each section with what your project actually uses. Other bundle skills (`/pair-next`, `/slice`, `/spec-prepare`) load this file on demand to ground their suggestions. The `verifier` subagent reads the **verification recipe** below.

## Branches

**One branch per Spec** (TEP-0010), named `spec/SP-{n}`:

```
spec/SP-3
spec/SP-7
```

Every slice under the Spec lands as **commits on that one branch** — there is **no per-slice branch**. The branch opens when the Spec's first slice starts and closes when the Spec is accepted (its one PR merges). When specs run in parallel, the branch is a **worktree** (TEP-0008) that lives for the whole Spec and is retired at merge — not per-slice.

Branches go to `main` via the Spec's single PR. Long-lived feature branches are an exception, not the default.

## Pull requests

**One PR per Spec** (TEP-0010) — never one per slice. The PR carries the whole Spec and merges at acceptance.

- Title: `<spec title>` (matches `specs/SP-{n}/spec.md`'s `# heading`).
- Body opens with a one-line summary + the Spec it closes (`Closes SP-3`), then lists the slices it delivers (`SP-3_SL-1 … SP-3_SL-4`).
- Description includes:
  - What changed across the slices (a short paragraph).
  - Files touched (auto-generated by `gh pr create` is fine).
  - How to verify (the verification recipe below; any manual steps).
- The PR is **merged by the acceptance step** (`/pair-next` runs `gh pr merge spec/SP-{n}` after the human accept) — not hand-merged mid-Spec.
- Reviewers: per the team's CODEOWNERS or the user's preference.

## Commits

Conventional Commits-ish but flexible. Each **slice** is one (or a few) commits on the Spec branch — tag the slice handle so the branch's history reads as the slice trail:

```
<type>(<scope>): <one-line summary>

<optional paragraph>

Refs SP-3_SL-4
```

A common shape is one commit per slice titled `SP-3_SL-4: <slice title>`. Types in active use: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `build`. Scope is optional.

## Verification recipe (the `verifier` reads this)

The `verifier` subagent runs the commands below to gate a slice's move to **Done**. There is **no separate test-suite assumption** — the recipe is the source of truth. This is the Antora **docs-platform** repo, so the recipe is the docs build:

- Docs build: `npm run build`

`npm run build` runs Antora over `antora-playbook.yml`; it exits **non-zero** on a broken `include::`, a bad `xref:`, or any AsciiDoc error, because the playbook sets `runtime.log.failure_level: warn`. So a doc edit that breaks the site is a verification fail (TEP-tgh6iy — no green = not Done). A clean build (exit 0) is the green.

## Things to never do

- `git push --force` to `main`.
- `rm -rf` against `.git/`, `node_modules/`, or anywhere outside the workspace root.
- Skip `pre-commit` / `pre-push` hooks via `--no-verify`. If a hook fails, fix the underlying issue.
- Commit secrets. The `ThinkubeStore.writeFile` secret-scan catches the obvious patterns, but eyes on the diff before pushing remains the human's responsibility.
